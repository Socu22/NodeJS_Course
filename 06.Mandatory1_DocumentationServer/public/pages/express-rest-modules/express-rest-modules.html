REST API Conventions – HTTP verbs, CRUD endpoints, plural nouns


how a rest api endpoints universally look like / follows established conventions: This is the order of HTTP verb. In the right order. The noun of the data has to be in plural as it reflects the backends table.
GET /examples → Retrieve all examples
GET /examples/{id} → Retrieve a specific example by ID
POST /examples → Create a new example (data in body)
PUT /examples/{id} → Fully update/replace an example by ID
PATCH /examples/{id} → Partially update a example by ID
DELETE /examples/{id} → Delete a example by ID

ES Modules vs CommonJS – Import/export syntax, live bindings, top-level await
ES Modules (ECMAScript Modules) it is modern way to organize code into separate file scripts with import & export Functionallity. The old way was commonjs. As ES modules are newer, there are intergrated Asynchronous instead of Synchronous loading time. Each of the Modules has Always been its own scope, in commonjs, it is cached, in ES modules on top of something similiar to caching, it has live bindings which binds everything which imports the module to its live changes, which commonjs doesn't as it only makes a cache when it requires it one time. At the same time Top-level await is in ES modules, which means you can say 'await' for a response when fetching data / when sending a Request and awaiting a response outside of a function. It bypasses one of the limitation of its javascript foundation as well as commonjs.
